# 参与贡献 Skill Pilot

感谢你为 **Skill Pilot** 做出贡献。

本指南说明了推荐的 Git 工作流，适用于以下用户：

1. 通过官方安装脚本安装 Skill Pilot
2. 在本地自己的分支上进行开发
3. 将仓库 fork 到自己的 GitHub 账号用于备份
4. 向 `contrib` 分支发起一个干净的 pull request

---

# 仓库分支

当前仓库使用以下分支：

- `codeware` —— 默认分支
- `contrib` —— 贡献目标分支

重要说明：

- 安装脚本会让用户从默认的 `codeware` 分支开始。
- 若要创建一个**干净的贡献 pull request**，请从 `upstream/contrib` 创建新的功能分支。

---

# 1. 安装 Skill Pilot

用户可以使用以下命令安装 Skill Pilot：

```bash
curl -fsSL https://skill-pilot.ai/install.sh | bash
```

这会执行类似如下的命令：

```bash
git clone https://github.com/X-School-Academy/skill-pilot
cd skill-pilot
git checkout -b user
```

此时：

* 本地仓库是从官方仓库克隆下来的
* 当前分支是 `user`
* `user` 分支是从默认分支 `codeware` 创建出来的

检查当前分支：

```bash
git branch
```

检查远程仓库：

```bash
git remote -v
```

通常在刚 clone 完之后，官方仓库就是 `origin` 远程仓库。

---

# 2. 在 `user` 分支上开发

用户可以在 `user` 分支上进行本地修改：

```bash
git status
git add .
git commit -m "My local changes"
```

常用命令：

```bash
git branch
git status
git log --oneline --decorate --graph -n 10
```

---

# 3. Fork 仓库并连接个人 GitHub 远程仓库

开始本地开发后，用户可以在 GitHub 上 fork 这个仓库。

打开：

```text
https://github.com/X-School-Academy/skill-pilot
```

点击 **Fork**。

推荐的 fork 方式：

* 只 fork 默认分支（`codeware`）

这样会创建一个类似下面的 fork：

```text
https://github.com/<your-github-user>/skill-pilot
```

## 更新本地远程仓库配置

由于本地仓库最初是从官方仓库克隆的，所以 `origin` 通常指向官方仓库。

我们建议先把这个远程仓库重命名为 `upstream`，然后把用户自己的 fork 添加为 `origin`。

```bash
git remote rename origin upstream
git remote add origin https://github.com/<your-github-user>/skill-pilot.git
```

再次检查远程仓库：

```bash
git remote -v
```

预期结果：

```bash
origin    https://github.com/<your-github-user>/skill-pilot.git (fetch)
origin    https://github.com/<your-github-user>/skill-pilot.git (push)
upstream  https://github.com/X-School-Academy/skill-pilot.git (fetch)
upstream  https://github.com/X-School-Academy/skill-pilot.git (push)
```

## 将 `user` 分支推送到个人 fork

这有助于把工作保存到 GitHub：

```bash
git checkout user
git push -u origin user
```

`-u` 的含义：

* `-u` 表示 `--set-upstream`
* 它会告诉 Git：本地 `user` 分支要跟踪 `origin/user`

之后，当你位于 `user` 分支时，通常以下命令都能直接使用：

```bash
git push
git pull
```

检查跟踪信息：

```bash
git branch -vv
```

# 4. 从 `contrib` 创建干净的贡献分支

当你准备把代码贡献回官方仓库时，**不要**直接使用 `user` 分支来向 `contrib` 发起 pull request。

正确做法是：

1. 获取官方仓库的最新分支
2. 从 `upstream/contrib` 创建一个新的功能分支
3. 把你的改动迁移或重新应用到这个功能分支上
4. 将该功能分支推送到你的 fork
5. 从你的 fork 向 `X-School-Academy/skill-pilot:contrib` 发起 pull request

## 获取官方仓库的最新分支

```bash
git fetch upstream
```

检查远程分支：

```bash
git branch -r
```

你应该会看到类似结果：

```bash
origin/user
upstream/codeware
upstream/contrib
```

## 创建贡献功能分支

示例：

```bash
git checkout -b my-feature upstream/contrib
```

这会基于官方的 `contrib` 分支创建一个新的本地分支。

---

# 5. 将你的改动带入贡献分支

有几种方式可以做到这一点。

## 方式 A：手动重新应用改动

如果改动很小，你可以复制文件或重新编辑，然后提交：

```bash
git add .
git commit -m "Add my contribution"
```

## 方式 B：从 `user` 中 cherry-pick 提交

如果你在 `user` 分支上的工作已经提交过，而且提交很干净，你可以直接 cherry-pick 这些提交。

先查看提交：

```bash
git log --oneline user
```

然后 cherry-pick 选中的提交：

```bash
git cherry-pick <commit-id>
```

示例：

```bash
git cherry-pick abc1234
```

如果有需要，也可以一次 cherry-pick 多个提交：

```bash
git cherry-pick <commit1> <commit2> <commit3>
```

这通常是最干净的方式。

---

# 6. 将贡献分支推送到个人 fork

将功能分支推送到用户自己的 fork：

```bash
git push -u origin my-feature
```

之后，本地分支就会跟踪远程 fork 上对应的分支。

检查跟踪情况：

```bash
git branch -vv
```

---

# 7. 向官方 `contrib` 分支发起 pull request

在 GitHub 上打开用户自己的 fork：

```text
https://github.com/<your-github-user>/skill-pilot
```

然后按以下设置创建 pull request：

* **base repository**：`X-School-Academy/skill-pilot`
* **base branch**：`contrib`
* **head repository**：`<your-github-user>/skill-pilot`
* **compare branch**：`my-feature`

然后提交 pull request。

重要说明：

* base 必须是 `contrib`
* head 分支应当是你从 `upstream/contrib` 创建出来的干净功能分支

# 理解 `origin` 和 `upstream`

在本指南中：

* `upstream` = Skill Pilot 官方仓库
* `origin` = 用户自己的 fork

推荐配置：

```bash
origin    -> https://github.com/<your-github-user>/skill-pilot.git
upstream  -> https://github.com/X-School-Academy/skill-pilot.git
```

这样工作流会更容易理解：

* 从 `upstream` 拉取官方更新
* 将个人工作推送到 `origin`

# 如何检查哪个分支跟踪了哪个远程分支

使用：

```bash
git branch -vv
```

示例：

```bash
* user       abc1234 [origin/user] local work
  my-feature def5678 [origin/my-feature] add docs
  codeware   123abcd [upstream/codeware] sync official
```

这表示：

* 本地 `user` 跟踪 `origin/user`
* 本地 `my-feature` 跟踪 `origin/my-feature`
* 本地 `codeware` 跟踪 `upstream/codeware`

也可以直接检查某个分支：

```bash
git config --get branch.user.remote
git config --get branch.user.merge
```

# 不推送也设置分支跟踪远程分支

如果远程分支已经存在，你也可以手动设置跟踪关系：

```bash
git branch --set-upstream-to=origin/user user
```

示例：

```bash
git branch --set-upstream-to=upstream/codeware codeware
```

检查结果：

```bash
git branch -vv
```

# 故障排查

## 我误用了 `orgin` 这个名字去 fetch

如果你把远程仓库名拼错成了 `orgin`，Git 会把远程分支记录到 `orgin/...` 下，而不是 `origin/...`。

检查远程仓库：

```bash
git remote -v
```

将其重命名：

```bash
git remote rename orgin origin
```

然后重新 fetch：

```bash
git fetch origin
```

## `fatal: the requested upstream branch 'origin/codeware' does not exist`

通常以下几种情况之一成立：

1. 远程仓库名写错了，例如把 `origin` 写成了 `orgin`
2. 远程分支还没有 fetch 下来
3. 远程分支本身不存在

检查：

```bash
git remote -v
git branch -r
```

然后使用正确的远程跟踪分支名称。

## 我的 pull request 包含了太多无关改动

这通常意味着该分支是从 `codeware` 创建的，但 pull request 的目标却是 `contrib`。

修复方法：

1. 获取官方仓库最新内容
2. 从 `upstream/contrib` 创建一个新的功能分支
3. 只 cherry-pick 需要的提交
4. 推送这个干净的分支
5. 重新打开 pull request

示例：

```bash
git fetch upstream
git checkout -b my-feature-clean upstream/contrib
git cherry-pick <commit-id>
git push -u origin my-feature-clean
```

# 推荐的日常工作流

## 首次设置

```bash
curl -fsSL https://skill-pilot.ai/install.sh | bash
cd skill-pilot

git remote rename origin upstream
git remote add origin https://github.com/<your-github-user>/skill-pilot.git

git checkout user
git push -u origin user
```

## 持续保存个人工作

```bash
git checkout user
git add .
git commit -m "Update local work"
git push
```

## 准备贡献分支

```bash
git fetch upstream
git checkout -b my-feature upstream/contrib
git cherry-pick <commit-id>
git push -u origin my-feature
```

然后从这里发起 pull request：

```text
<your-github-user>/skill-pilot:my-feature
```

目标是：

```text
X-School-Academy/skill-pilot:contrib
```

# 总结

* 安装流程从 `codeware` 开始
* 本地开发可以在 `user` 分支进行
* 可以 fork 到个人 GitHub 作为备份
* 将官方仓库远程名改为 `upstream`
* 将个人 fork 添加为 `origin`
* 把 `user` 分支推送到个人 fork 以保存工作
* 贡献代码时，应从 `upstream/contrib` 创建干净的功能分支
* 将该功能分支推送到个人 fork
* 向官方的 `contrib` 分支发起 pull request
